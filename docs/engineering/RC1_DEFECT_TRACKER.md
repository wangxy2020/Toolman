## 2. 严重度定义

| 级别 | 定义 | 响应 SLA | 示例 |
|------|------|----------|------|
| **P0** | 崩溃、数据丢失/ corruption、无法登录、P2P 完全不可用、Release 包无法安装启动 | 24h 内 ack，48h 内 fix 或 rollback | 启动即 crash；群消息全部丢失 |
| **P1** | 核心路径阻断，有临时 workaround | 3 工作日内 triage，RC1 结束前 fix 或 documented workaround | WAN 群无法加入但 LAN 可用；Hub 离线且无缓存提示 |
| **P2** | 体验、文案、非阻断功能瑕疵 | 记录排期，可带入 RC2 | 按钮对齐；搜索排序不理想 |
| **P3** | 建议 / 增强，非缺陷 | 不阻塞 RC | 快捷键偏好 |

**核心路径模块**（P0/P1 优先覆盖）：登录 · 对话 · 知识库 · 社区 · P2P 群组 · OTA · 崩溃上报

### 3.2 状态流转

```
open → triaged → in_progress → fixed → verified → closed
                      ↓
                  wontfix / duplicate / deferred_RC2
```

| 状态 | 含义 |
|------|------|
| `open` | 新报告，未 triage |
| `triaged` | 已确认严重度与负责人 |
| `in_progress` | 修复中 |
| `fixed` | 修复已合入，待 RC1 包验证 |
| `verified` | 报告人在新包上确认 |
| `closed` | 关闭 |
| `wontfix` | RC1 不修复（须说明原因） |
| `duplicate` | 重复，填「关联」列主 ID |
| `deferred_RC2` | 明确推迟到 RC2（通常 P2） |

### 3.3 环境列填写规范

- **通道**：`staging` / `dev`（RC: 混用 dev 包报 RC1 缺陷）
- **Hub**：`remote https://hub.toolman.app` / 离线
- **userData**：是否独立 RC1 profile（推荐 `Toolman-RC1`）
- **网络**：LAN / WAN+TURN / 纯离线
- **TURN**：有 / 无（WAN 问题必填）

## 5. 复现步骤写作要点

1. **从干净状态开始**：是否新装 / 新 userData / 是否登录特定账号类型（CN/Global）
2. **一步一结果**：每步写操作 + 当时界面/日志表现
3. **可重复性**：必现 / 偶现（_N 次中 M 次_）/ 仅一次
4. **P2P/WAN**：两台机器角色（A/B）、是否同网、TURN 是否配置
5. **时间线**：崩溃前最后 3 个操作

| 字段 | 值 |

**复现**
1. A、B 均安装 `Toolman-0.2.0-rc.1-arm64.dmg`，独立 userData

**期望**：TURN 中继下 60s 内加入成功  
**实际**：超时，LAN 同网段可复现成功  
**Workaround**：同一 Wi-Fi 下测试
